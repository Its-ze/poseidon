# Sub-GHz + RF Radios Module Audit

Scope: 18 .cpp + 9 .h, ~7,690 LOC total. Drivers (CC1101 / nRF24 / nRF52 / LoRa SX1262) + 17 feature modules + RMT pulse engine + decode + signal DB.

## Current state

The bucket implements four radio domains gated behind a single `radio_switch()` mutex in `src/radio.cpp:135` that tears down whatever stack was active before bringing the next one up. CC1101 lives on shared HSPI (CS=13, GDO0=5) via the bmorcelli ELECHOUSE fork — `cc1101_hw.cpp:42` hands the SD's `sd_get_spi()` instance to `setSPIinstance()` so the lib skips its own `SPI.begin()` (which would re-grab pins in peripheral mode and hang on the MISO-wait loop in `Reset()`). `cc1101_park_others()` (`cc1101_hw.cpp:12`) explicitly calls `gps_end()` before driving pin 13 OUTPUT/HIGH (CC1101 CS == GPS UART TX), and also forces SD CS=12, nRF24 CS=6, LoRa NSS=5 HIGH. Init checks `Init()` return + `getCC1101()` PARTNUM probe so a wedged bus or wrong hat fails loudly. Default OOK preset is the Flipper AM650 (RxBW=650, DRate=3.794, ASK mod 2) — Bruce-compatible. RMT TX/RX (`cc1101_rmt.cpp`) uses IDF 5.5 `rmt_tx.h` / `rmt_rx.h` at 1 MHz resolution; capture filter is `signal_range_min_ns=3*1000` (3 µs, Bruce-derived — `cc1101_rmt.cpp:157`), gap_us configurable; channels are allocated per-call and freed on exit so WiFi/BLE can reclaim RAM.

Sub-GHz features: scan (`subghz_scan.cpp:133`, ISR edge counter + tight poll capture + Princeton/CAME/NICE/Linear decode + S/R hotkeys + publishes to `g_subghz_last_cap`), replay (`subghz_replay.cpp:115`, picks `.sub` from SD then `cc1101_rmt_tx`), record (`subghz_record.cpp:67`, `cc1101_rmt_rx` 20s window, auto-decode, freq autoscan with `yield()`), broadcast (`subghz_broadcast.cpp:214`, category picker `/poseidon/signals/cars,pranks,tesla,home,custom` + baked signals from `subghz_signals_data.h`, ON-AIR badge), bruteforce (`subghz_bruteforce.cpp:34`, RCSwitch over CC1101 GDO0 — Came/Nice/Linear/Chamberlain/Holtek/Ansonic), spectrum (`subghz_spectrum.cpp:362`, bar/waterfall/oscilloscope, double-buffered M5Canvas, static 28.8 KB BSS ring), jammer (`subghz_jammer.cpp:18`, intermittent + full carrier, 20s wall-clock cap), jam-detect (`subghz_jam_detect.cpp:68`, 10s baseline + 15 dBm trigger over 5 samples + SD log + siren). Decoder library at `subghz_decode.cpp` covers Princeton/CAME/NICE/Linear plus a generic 1T/3T fallback (~196 LOC, ±30% tolerance).

nRF24 lives on the same HSPI (CS=6, CE=4). `nrf24_hw.cpp:11` parks SD/CC1101/LoRa-NSS HIGH and reuses `sd_get_spi()`. `nrf24_suite.cpp` (771 LOC) is the omnibus: promiscuous sniffer using the 6-pipe Goodspeed/Bastille noise-address table (`nrf24_suite.cpp:186-194`) — `EN_RXADDR=0x3F`, `SETUP_AW=0`, six 2-byte noise addresses matching Bruce verbatim; raw register writes go through `nrf_write_reg()` which beginTransaction on `sd_get_spi()` (the v0.6.1 Hydra cap fix — global FSPI was stealing the display's GPIO matrix). MouseJack injection picks from the sniffer's `s_devs[]` table, sets SETUP_AW=0x03, fires Logitech/MS HID frames; BLE-spam over nRF24 with ADV_NONCONN_IND + CRC24 + whitening (`ble_encode_packet`); 2.4 GHz scanner uses `testRPD()`; expanded jammer with 7 presets (BLE / WiFi 1/6/11 / Zigbee / Drone / HID), CW + data-flood modes, hops via the `stopConstCarrier→setChannel→startConstCarrier` cycle (v0.6.0 bug fix re-applied — bare `setChannel` does not actually move a const carrier).

LoRa SX1262 (`lora_hw.cpp:106`) brings the radio up against `sd_get_spi()` shared HSPI (NSS=5, BUSY=6, DIO1=4, RST=3) and drives the PI4IOE5V6408 antenna switch directly over I²C 0x43 (bypassing `M5.getIOExpander()` which LoadProhibited-panics on this board). Init uses the long-form `begin()` with TCXO=1.8 V + DC-DC regulator, errata 0x8B5 RX-sensitivity bit, and boosted-gain mode. Three feature surfaces: `radio_lora.cpp` (passive scan, GPS-tagged beacon TX, Meshtastic LongFast header decode, GPS-fix viewer); `lora_spectrum.cpp` (bar / waterfall / scope on a single tuned frequency — single-bin RSSI time-series rather than a frequency sweep, packet history ring of 6); ISR-driven RX via `setPacketReceivedAction()`.

nRF52840 Feather companion uses UART2 (despite header comments still saying "UART1") at 115200 baud on G3 TX / G4 RX (`nrf52_hw.cpp:18-26`). PING/PONG handshake, line-based command/response with OK/ERR sentinels and STOP-terminated streams. NeoPixel modes (24 enumerated in `nrf52_led.h`) sent as `LED:<id>` fire-and-forget. Feature menus call into Feather firmware: BLE 5.0 full scan, Coded-PHY long-range scan, BLE ADV sniffer + pcap, 802.15.4 Zigbee sniffer, Scout-and-Strike dual-radio chain (`nrf52_scout_strike.cpp`), full BLE MITM relay (`nrf52_ble_mitm_relay.cpp`), and WiFi-deauth → BLE-provision capture combo (`nrf52_wifi_ble_combo.cpp:51`). `rf_finder.cpp` thermometer for CC1101 RSSI and nRF24 `testRPD` strength bar.

## Defects

1. **CRITICAL — rf-001** `src/features/subghz_jam_detect.cpp:78,202-205` — feat enters `radio_switch(RADIO_SUBGHZ)` then on every failure or exit path returns WITHOUT calling `radio_switch(RADIO_NONE)`. Specifically: line 78 early-return after `cc1101_begin` failure, line 114 user-ESC during warmup, line 194 user-ESC during monitor. `s_active` stays `RADIO_SUBGHZ`, so a subsequent WiFi feature thinks "nothing to tear down" — GPS stays dead (was killed by enter-side `gps_end()` in radio.cpp:171), and the next CC1101 feature starts from `cc1101_park_others()` which short-circuits because `radio_switch` skips teardown when target == s_active. Cascading: next CC1101 reentry skips full re-park sequence. Fix: every exit path must call `radio_switch(RADIO_NONE)`; also remove the `pick_freq()` ESC return at line 73 (no radio cleanup either). Effort: S.
2. **CRITICAL — rf-002** `src/features/subghz_jam_detect.cpp:78` — when `cc1101_begin` fails the function returns without `radio_switch(RADIO_NONE)` AND without re-arming GPS. Pin 13 is left as the in-driver OUTPUT/HIGH state from `cc1101_park_others()`, GPS UART permanently broken until reboot. Fix: explicit `radio_switch(RADIO_NONE)` (which re-runs `gps_begin`). Effort: S.
3. **HIGH — rf-003** `src/features/subghz_jammer.cpp:35-39` — the 20 s safety cap fires ONLY when entering the `if (active && now - jam_start > JAM_MAX_MS)` block, but in mode 0 (intermittent) the loop spends 700-3000 µs per iteration in TX+idle waits — yes, fine, but in mode 1 (full carrier, `cc1101_set_tx` once at line 91) there is NO inner loop work — the main `while(true)` ticks every ~2 ms (delay at line 78). Cap fires correctly. However: the cap only calls `cc1101_set_idle()` — does NOT call `cc1101_end()` or `radio_switch(RADIO_NONE)`. Chip is left in idle state but still occupying the radio slot. Combined with an explicit ESC after the cap, the eventual `cc1101_end()` at line 96 runs fine — so this is not a bug but the cap-fire user-toast claims "20s limit" while the chip silently stays armed for re-ENTER. Minor UX. Effort: S.
4. **HIGH — rf-004** `src/cc1101_hw.cpp:91` — `cc1101_end()` calls `setSidle` + `goSleep` but does NOT reset GDO0 to INPUT or de-park CS lines 12/13/6/5. The next `cc1101_begin()` re-parks, but a sub-GHz → LoRa transition would see CC1101 CS (13) still driven HIGH (fine) but `cc1101_end` left the GDO0 pin in whatever state the previous feature ended in (TX path = OUTPUT from `subghz_broadcast.cpp:342`, replay `subghz_replay.cpp:165`, brute `subghz_bruteforce.cpp:80` indirect through ELECHOUSE). LoRa-side doesn't drive 5, but if user re-enters CC1101 scan, GDO0=OUTPUT will be silently re-INPUT'd at `cc1101_hw.cpp:84`. Net: latent, currently harmless. Fix: add `pinMode(CC1101_GDO0, INPUT)` to `cc1101_end()`. Effort: S.
5. **HIGH — rf-005** `src/lora_hw.cpp:118-124` — `lora_begin` parks SD CS=12 HIGH but does NOT park CC1101 CS=13 or nRF24 CS=6. The comment says "CC1101 never coexists with LoRa (radio_switch handles mutual exclusion)" — true at the s_active level, but if a feature exited via early-return WITHOUT calling `radio_switch(RADIO_NONE)` (see rf-001), CC1101 CS could be left driven from `cc1101_park_others()` in a way that the LoRa radio sees noise. Defensive park-all would cost nothing here. Fix: park CS=13 (after `gps_end` check — pin shared with GPS TX) and CS=6 HIGH. Effort: S.
6. **HIGH — rf-006** `src/features/nrf24_suite.cpp:32-42` — `nrf_write_reg()` does raw SPI register writes on `sd_get_spi()`. Good — fixed in v0.6.1. BUT: the call sites at lines 186-187 (sniffer init: `EN_RXADDR=0x3F`, `SETUP_AW=0`) AND line 359 (mousejack: `SETUP_AW=0x03`) happen AFTER `nrf24_begin()` which has already opened pipes/configured the radio through the RF24 lib. The lib internally caches CONFIG/EN_AA shadow state; bypassing it leaves a divergence (`SETUP_AW=0` then `setAutoAck(false)` is fine, but `setAutoAck(true)` in mousejack at line 355 followed by `nrf_write_reg(0x03, 0x03)` does not update the shadow). Subsequent RF24 API calls reading from the lib's cached state may write inconsistent values back to the chip. Currently masked because mousejack uses lib's `setRetries/openWritingPipe/setChannel/write` which re-issue commands. Latent. Fix: prefer `rf.setAddressWidth(2)` / `rf.setAddressWidth(5)` if RF24 lib supports it (it does — used at `nrf24_suite.cpp:462`), eliminate the raw write at 187/359. Effort: S.
7. **HIGH — rf-007** `src/features/subghz_replay.cpp:67` — `pick_sub_file` does `SD.open("/poseidon")` and recurses ONE level via `f.path()` lookup, but actually it only iterates root-level `.sub` files in `/poseidon/` itself — there is no subdir descent. Real recordings live under `/poseidon/signals/custom/`. So this feature has been silently empty for users who only have recordings from `subghz_scan` or `subghz_record` (which write to `/poseidon/signals/custom/`). Fix: descend into `/poseidon/signals/` or pivot to the `subghz_broadcast.cpp` category picker. Effort: S.
8. **HIGH — rf-008** `src/features/subghz_record.cpp:175` — `cc1101_rmt_rx` is invoked with `timeout_ms=20000` (20 s) which exceeds the FreeRTOS TWDT (default 5 s in IDF 5.5). RX path blocks on `ulTaskNotifyTake` for the full 20 s if no signal arrives. There is no `yield()`/`esp_task_wdt_reset()` inside `cc1101_rmt_rx`. Idle-task watchdog may panic. The `yield()` in `subghz_record.cpp:159` is in the autoscan branch only. Fix: split the 20 s wait into 1 s slices with `yield()` between, OR disable WDT for the recording window, OR document that the user-perceived hang IS the design. Effort: M.
9. **MEDIUM — rf-009** `src/features/subghz_jammer.cpp` and `subghz_bruteforce.cpp` — neither feeds the watchdog in their tight TX loops. Brute does `delay(2)` per code (line 106) and TX is fast, so within budget for 9-12 bit protos. Jammer mode-0 hits the loop at 700-3500 µs cycle which is also fine. But mode-1 full-carrier path has nothing between the initial `cc1101_set_tx` and the 20 s cap — main loop only `delay(active ? 2 : 20)`. Should be fine but is fragile if the cap path is ever extended. Fix: explicit `esp_task_wdt_reset()` once per second in active loops. Effort: S.
10. **MEDIUM — rf-010** `src/features/subghz_record.cpp:90` — `ui_rssi_scope_reset()` called but `ui_rssi_scope` history is shared global state. Other features touching this scope (any future caller) leaks history between sessions. Defensive but not currently exposed. Effort: S.
11. **MEDIUM — rf-011** `src/features/nrf24_suite.cpp:684,748` — when the 20 s jammer cap fires, the code does `rf.powerDown(); ui_toast(...); nrf24_begin();` — full re-init. But `nrf24_begin` calls `nrf24_park_others()` which drives SD CS=12 HIGH; that's normally fine, but the user-ESC path at line 748 issues `rf.stopConstCarrier(); nrf24_begin();` then immediately `break;` out of the outer loop, which calls `nrf24_end()` at line 769. The intermediate re-begin is wasted work. Cosmetic. Effort: S.
12. **MEDIUM — rf-012** `src/nrf52_hw.cpp:24-26,46` — header comment says "UART1 (G3 TX, G4 RX)" but `nrfSerial(2)` (UART2) and `NRF52_UART_RX=4, TX=3`. Also `Serial.println` at line 46 says "UART1 (G3/G4)". Documentation drift only — pin defines are right. Effort: S.
13. **MEDIUM — rf-013** `src/features/lora_spectrum.cpp:196,234` — waterfall mode `malloc(60*200*2 = 24000 B)` of `uint16_t ring`. On entry after radio/wifi/SD have churned heap, this allocation can fail (`ring=NULL` falls through to `ui_toast("OOM")` then returns WITHOUT calling `lora_end()` or `radio_switch(RADIO_NONE)`. Worse than rf-001: leaks the LoRa radio + antenna switch ON. Same teardown gap at line 234 on user ESC — the `free(ring)` happens but no `lora_end` / `radio_switch`. The outer `feat_lora_spectrum` does end the radio (line 387) which catches it — OK on user-ESC. But OOM path skips back to the menu picker which `continue`s the mode-picker loop without recovery. Fix: return from outer feature on inner failure, OR move ring alloc to static. Effort: S.
14. **MEDIUM — rf-014** `src/features/lora_spectrum.cpp:241-307` — scope mode does `radio.standby(); radio.setFrequency(freq); radio.startReceive();` on every freq change. Each transition costs ~10 ms on SX1262. Holding +/- triggers a tight `while(millis()-t<40)` loop at line 299-305 that can mash setFrequency dozens of times per second. Could wedge on BUSY. Add a freq-change debounce (≥100 ms). Effort: S.
15. **MEDIUM — rf-015** `src/cc1101_hw.cpp:12-27` — `cc1101_park_others()` always calls `gps_end()` and toggles pin 13 HIGH, but `cc1101_begin` then calls `cc1101_park_others()` again on every freq change scenarios via `cc1101_set_freq` — actually no, `cc1101_set_freq` doesn't park. But `subghz_scan.cpp:127` calls `cc1101_begin(freq)` after save_sub — this fires gps_end+park unnecessarily. Cheap call but each fires `gps_end()` which is destructive (frees driver). ### vs PORKCHOP (0ct0sec, 800f06b)

PORKCHOP's only RF radio is the SX1262 LoRa, which it **only quiesces** for SD bus coexistence — `config.cpp:382 prepareCapLoraGpio` resets G13 then SX1262 reset + CS HIGH so the chip tri-states MISO. No LoRa stack. PORKCHOP's BLE spam runs on NimBLE/ESP32 BLE (`piggyblues.cpp`), not CC1101 or nRF24. Comparison points:

- **SX1262 quiesce on session-switch**: POSEIDON does park LoRa NSS HIGH in `cc1101_park_others()` (cc1101_hw.cpp:26) and `nrf24_park_others()` (nrf24_hw.cpp:18) but does NOT issue a SX1262 reset pulse + CS HIGH like PORKCHOP. With the LoRa hat connected and the SX1262 in any state other than sleep, it could drive MISO during CC1101/nRF24 ops. `lora_end()` (`lora_hw.cpp:211`) does call `s_radio->sleep()` first which puts SX1262 in deep-sleep (MISO Hi-Z). But if a feature returns without calling `lora_end()` (rf-013 OOM path), the next CC1101 feature has a chatty MISO line. **Mitigation:** add `pinMode(LORA_RST, OUTPUT); digitalWrite(LORA_RST, LOW); delay(1); HIGH;` to `cc1101_park_others` for defense-in-depth. Backlog candidate.
