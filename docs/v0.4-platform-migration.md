## 1. What's broken and why (don't re-investigate this)

POSEIDON's deauth frames return `ESP_ERR_INVALID_ARG (0x102, 258)` from
`esp_wifi_80211_tx` on every call. Every deauth feature in the codebase
is affected: `feat_wifi_deauth`, `wifi_clients`, `wifi_clients_all`,
`wifi_deauth_extras`, `triton`.

**Root cause:** stock `espressif32@6.7.0` bundles ESP-IDF 5.3's WiFi
blob (`libnet80211.a`), which filters management frame subtypes `0xC`
(deauth) and `0xA` (disassoc) regardless of:
- WiFi mode (STA, AP, AP_STA all blocked)
- Interface used for TX (WIFI_IF_STA and WIFI_IF_AP both blocked)
- addr2 matching the interface MAC (confirmed by spoofing — still rejected)
- Promiscuous mode on/off
- `en_sys_seq` true/false

**What does work:** subtype `0x8` (beacon) TX works on stock — so the
blob allows MGMT, just not deauth/disassoc specifically. This is a
subtype-specific block introduced in IDF 5.3.

**The fix:** migrate to pioarduino + Bruce's (`bmorcelli`) patched libs
for IDF 5.5. The patched `libnet80211.a` has the subtype filter NOP'd.
Bruce (M5-family firmware, actively maintained, sibling project) uses
this exact stack and deauth works. This is the same workaround
ESP32Marauder uses in newer versions.

### Evidence (if future-you is doubting the diagnosis)

Previous session logged from serial:
```
[deauth] set_mac 8A:D5:9D:5F:AF:19 rc=0
[80211_tx] deauth rc=258 (0x102) sta_mac=8A:D5:9D:5F:AF:19 addr2=8A:D5:9D:5F:AF:19
```

MAC spoof succeeded, addr2 matches sta_mac, still rejected. Beacon spam
(subtype 8) works in the same environment with no changes. Bruce uses
the same POSEIDON-style frame bytes verbatim (verified against
`ESP32Marauder/esp32_marauder/WiFiScan.cpp:8033` `sendDeauthFrame`) and
frames land. Bruce's `platformio.ini` points at pioarduino + Bruce's
lib zip. That's the entire difference.

## 3. Why the previous attempt failed (don't repeat these)

Previous session tried from Git Bash on Windows. Failures encountered:

1. **MSys detection blocks toolchain install.** PlatformIO runs
   `idf_tools.py` during new-platform install. idf_tools.py hard-fails
   with "ERROR: MSys/Mingw is not supported" because pioarduino's
   install scripts see Git Bash's MSys environment. Setting
   `MSYSTEM=""` and `MINGW_PREFIX=""` did NOT help — the detection
   checks multiple environment markers.

2. **504 on stock Espressif lib URL.**
   `https://github.com/espressif/arduino-esp32/releases/download/3.3.6/esp32-core-3.3.6-libs.tar.xz`
   returned 504 consistently. Resolved after adding the `platform_packages`
   override pointing at Bruce's patched zip — PlatformIO no longer
   attempted the upstream download once the override was in place.

3. **Manual toolchain extraction didn't propagate.** Previous session
   extracted `xtensa-esp-elf-14.2.0_20251107-x86_64-w64-mingw32.zip`
   directly into
   `C:\Users\D\.platformio\packages\toolchain-xtensa-esp-elf\` and
   verified binaries ran when invoked directly. But PlatformIO's build
   subprocess didn't inherit an updated PATH — kept erroring with
   `'xtensa-esp32s3-elf-g++' is not recognized`. Manual `export PATH="..."`
   in the parent shell helped for a single run but was fragile.

4. **Incomplete build with "SUCCESS" false positive.** Final attempt
   reported exit 0 and "Successfully created ESP32-S3 image" but no
   `firmware.bin` was produced. Background task reporter lies on this
   path. Always `ls .pio/build/cardputer/firmware.bin` before believing.

### The root cause of all 4: Git Bash + MSys + PlatformIO install machinery

**Recommendation: use WSL2 for this migration.** Poseidon's Kali WSL2
distro is already set up (used for the bughunt pipeline). Install
PlatformIO inside Kali, work from there, no MSys issues. See section 4.

## 4. The migration (execute in this order)

### Prerequisites
- Access to `C:\Users\D\poseidon` repo (clean tree on master)
- WSL2 Kali distro running (POSEIDON is stored on the Windows side; WSL
  can access it via `/mnt/c/Users/D/poseidon`)
- Internet connectivity (platform zip + toolchain + libs are each ~100MB)
- Cardputer connected (for the final flash test)

### Step 1: Enter WSL and verify stable build works there first

```bash
# From Windows, launch Kali WSL
wsl -d kali-linux

# Inside WSL
cd /mnt/c/Users/D/poseidon
# Install pio if not already
pip install -U platformio
# Confirm stable compiles in WSL (sanity — shouldn't fail since it's pure toolchain)
pio run
# Expect: [SUCCESS], firmware.bin in .pio/build/cardputer/
ls -la .pio/build/cardputer/firmware.bin
```

If this fails, do not proceed — fix the WSL build env first.

### Step 2: Back up current stable config (already done, verify)

```bash
ls -la /mnt/c/Users/D/poseidon/platformio.ini.stable
# If it doesn't exist, create from current:
cp /mnt/c/Users/D/poseidon/platformio.ini /mnt/c/Users/D/poseidon/platformio.ini.stable
```

### Step 3: Swap platform config

Edit `platformio.ini`. Change:

```ini

board                 = m5stack-stamps3
```

To:

```ini
; pioarduino 55.03.36 + Bruce's patched WiFi libs. Migration doc:
;   docs/v0.4-platform-migration.md
; Rollback: cp platformio.ini.stable platformio.ini

platform_packages     = framework-arduinoespressif32-libs @ https://github.com/bmorcelli/esp32-arduino-lib-builder/releases/download/idf-release_v5.5/bruce_esp32-arduino-libs-20260123-153546.zip
board                 = m5stack-stamps3
```

Leave everything else (`framework`, `build_flags`, `lib_deps`, etc.) unchanged.

### Step 4: Install the new platform

```bash
cd /mnt/c/Users/D/poseidon
pio pkg install
# Expect: downloads ~300MB of tooling, finishes clean, no errors about
# MSys. On WSL you get normal Linux toolchain binaries which just work.
```

If MSys errors re-appear in WSL, check that you launched WSL fresh and
aren't accidentally in a mix-mode shell.

### Step 5: Compile test (no flash)

```bash
rm -rf .pio/build
pio run 2>&1 | tee /tmp/pio_v04.log
tail -5 /tmp/pio_v04.log
ls -la .pio/build/cardputer/firmware.bin
```

Expect "SUCCESS" at the end AND a firmware.bin file. **Both conditions
required** — the background-task exit codes have lied in the past.

### Step 6: Expected compile warnings (ignore these)

GCC 14 (in the new toolchain) is stricter than GCC 8 (in the old one).
Expect warnings like:

- `-Wvolatile` deprecation on `s_var++` where `s_var` is `volatile uint32_t`
  — appears in `net_responder.cpp`, `c5_cmd.cpp`, several features
- `-Wformat-truncation` on some `snprintf` paths with `%d` in `menu.cpp:634`
- `POSEIDON_VERSION` redefined warning from the build flag colliding with
  a #define in `src/app.h:29`

These are all warnings, not errors. Don't fix them during migration —
fix in a follow-up commit once the platform is live.

### Step 7: Errors that might surface + fixes

If the compile actually errors (not just warns), the most likely causes:

**A. `M5Cardputer` API drift.** Core 3.3.6 uses M5Unified 0.2.x but some
internal APIs shifted. If you see errors in `src/main.cpp` around
`M5.config()` or `M5Cardputer.begin()`, check the M5Unified version
being resolved — may need to bump `m5stack/M5Unified` in `lib_deps`.

**B. `NimBLE-Arduino` version mismatch.** NimBLE 1.4.x was for Core 3.1;
Core 3.3.6 may want NimBLE 2.x. Bump `h2zero/NimBLE-Arduino` to `^2.0` if
BLE features fail to compile. Check BLE feature files for API changes —
NimBLE 2.x renamed several constants.

**C. `RadioLib` version.** RadioLib 7.x should be fine, but if LoRa
features fail, check for renamed SX1262 methods. Bump to `^7.2.0`.

**D. `esp_wifi.h` / `esp_event.h` API drift.** Unlikely for raw 80211_tx
paths (they're stable across IDF versions) but promiscuous callback
signatures may have changed. Double-check `wifi_promiscuous_pkt_t` usage
in `wifi_pmkid.cpp`, `wifi_clients.cpp`, `wifi_clients_all.cpp`.

**E. Partition layout.** The new platform defaults may differ. The
stable config uses `default_8MB.csv` — keep that line, should work.

**Fix strategy:** if any of A–E hits, DON'T revert yet. Fix the error
in place, verify other features still work, commit the fix as a
pre-migration preparation. Each library bump is safer as its own PR.

### Step 8: Flash to Cardputer

Only after Step 5 succeeds AND produces a firmware.bin:

```bash
# From WSL, Cardputer is at /dev/ttyACM0 usually (or similar)
pio device list
# Identify the COM device. If on COM16 under Windows, WSL2 sees it as
# /dev/ttyS16 or similar — USBIPD-WIN forwarding may be needed. If so,
# flash from Windows cmd.exe after compile on WSL finishes.

pio run -t upload --upload-port /dev/ttyACM0
# Or from Windows:
cmd.exe //c "pio run -t upload --upload-port COM16 -d C:\\Users\\D\\poseidon"
```

### Step 9: On-device regression test

Boot the device. Check in order:
1. Splash screen appears, boot jingle plays, main menu draws
2. WiFi Scan — finds APs (proves WiFi subsystem came up)
3. Bluetooth Scan — finds BLE devices (proves NimBLE works)
4. LoRa Scan — spectrum displays, no crash (proves RadioLib + M5 display work)
5. SD card — browse files (proves SD + FS layer work)
6. Any crash at boot = revert, diagnose from panic trace

### Step 10: The actual deauth test

1. WiFi → Scan → pick a non-PMF WPA2 AP you own
2. Enter → D (targeted deauth)
3. Watch serial: `[deauth] silent_ap ch=N rc=0` should appear
4. Watch UI counter: `sent:` should climb, `drops:` should stay near 0
5. Target device (phone/laptop connected to the AP) should lose WiFi

If any of these fails, serial logs tell the story. Previous session's
`[80211_tx]` error logger is still in `wifi_deauth_frame.h` — it'll
print `rc=N (0xNNNN)` if frames are rejected. Expect rc=0 with the
patched blob.

## 6. What NOT to do during migration

- **Don't rewrite the deauth code** — `wifi_deauth_frame.h` and all
  5 callers are already correct. The Marauder pattern is in place.
- **Don't swap libs without swapping platform** — Bruce's IDF 5.5 libs
  won't link against Core 3.1.0. They MUST go together.
- **Don't skip the compile test** — exit codes lie on this codebase.
  Always `ls .pio/build/cardputer/firmware.bin` to verify.
- **Don't flash from a failed compile** — you'll brick the device
  temporarily. The previous session bricked once and had to recover via
  usb-reset esptool flags.
- **Don't do this migration from Git Bash on Windows.** Seriously, use
  WSL. The previous session spent hours on MSys issues that simply
  don't exist on Linux.

## 8. Meta: this migration vs. doing it differently

Alternative paths that were considered and rejected in the prior session:

- **Patch only the WiFi libs without platform swap:** Bruce's IDF 5.5
  libs won't ABI-link against Core 3.1.0's ESP-IDF internals. Symbol
  signatures changed between 5.3 and 5.5. Abandoned.
- **Downgrade platform to espressif32@5.x (IDF 4.4):** Would match
  older Marauder precompiled libs, but NimBLE / M5Cardputer / RadioLib
  current versions don't support Core 2.x anymore. Too much library
  regression.
- **Build our own patched libnet80211 from scratch:** Requires
  esp32-arduino-lib-builder, which itself requires full ESP-IDF source.
  Binary blob patching takes hours, multiple hardware-specific tests.
  Not worth unless Bruce stops maintaining.
- **Route all deauth through ESP32-C5 companion:** Works if the user
  has a C5 node flashed with patched firmware. Doesn't require any
  Cardputer firmware changes. A solid FALLBACK but requires second
  hardware.

The platform migration is the single best long-term answer because:
- pioarduino is actively maintained
- Bruce's patched libs are kept up-to-date (latest zip dated 2026-04-07)
- Core 3.3.x is a minor version bump from 3.1.x (API drift is small)
- Everything else in POSEIDON is already tested on modern Core

## 11. Execution log (what actually shipped, 2026-04-18)

Real-world results of running this migration from WSL2 Kali:

### Deviations from the plan

**A. Target version bumped 55.03.36 → 55.03.38.** The stock Espressif
`esp32-core-3.3.6.tar.xz` tarball that pioarduino 55.03.36 tries to fetch
returned permanent 504 from GitHub's release CDN. Retries across 30+
minutes all failed. 55.03.38 (Arduino 3.3.8 / IDF 5.5.4) was published
on 2026-04-08 with a clean tarball and pulls fine. ABI-wise it's still
IDF 5.5 so Bruce's libs from the same idf-release_v5.5 tag link without
issue. Config ended up:

```ini
platform              = https://github.com/pioarduino/platform-espressif32/releases/download/55.03.38/platform-espressif32.zip
platform_packages     = framework-arduinoespressif32-libs @ https://github.com/bmorcelli/esp32-arduino-lib-builder/releases/download/idf-release_v5.5/bruce_esp32-arduino-libs-20260407-140520.zip
```

**B. WSL prereqs.** Fresh WSL2 Kali needed:
```bash
sudo apt install -y python3-venv python3.13-venv rsync
pip install --user --break-system-packages platformio
```
Without `python3-venv`, pioarduino's first `pio pkg install` creates a
broken penv and fails with "uv installation via pip failed" — had to
`rm -rf ~/.platformio/penv` and retry. `rsync` is used by
`scripts/wsl_sync_and_build.sh` and is not in the base Kali image.

**C. Source must be synced to WSL-native filesystem.** Building directly
from `/mnt/c/...` is ~50× slower due to 9P protocol overhead on every
stat/open. A no-op dependency scan took >5 min on /mnt/c vs <30 s after
rsync'ing to `~/poseidon_build`. See `scripts/wsl_sync_and_build.sh` for
the pipeline: rsync (excluding .pio/.git/scripts/docs) → `pio run` →
copy `.pio/build/cardputer/firmware.bin` back to /mnt/c.

### Source-level fixes needed for Core 3.3.8 / IDF 5.5.4

Three compile errors surfaced; none were in deauth code. All IDF API
drift, all one-line fixes:

1. **`src/c5_cmd.cpp:138`** — ESP-NOW recv callback signature changed in
   IDF 5.x. Old: `void cb(const uint8_t *mac, const uint8_t *data, int len)`.
   New: `void cb(const esp_now_recv_info_t *info, const uint8_t *data, int len)`.
   Source MAC is `info->src_addr`.

2. **`src/mesh.cpp:45`** — same ESP-NOW callback change.

3. **`src/mesh/meshtastic_node.cpp:63`** — `esp_read_mac` and
   `ESP_MAC_WIFI_STA` moved to `esp_mac.h` in IDF 5.x. Added
   `#include <esp_mac.h>`.

No library version bumps required — NimBLE-Arduino 1.4.1, RadioLib 7.4.0,
M5Cardputer 1.1.1, M5Unified 0.2.13 all compile clean against Core 3.3.8.

- RAM:   48.2% (157996 / 327680 bytes) — plenty of headroom
- Clean compile, zero warnings from POSEIDON source, zero deprecations
- `.pio/build/cardputer/firmware.bin` — 2092048 bytes
